The auth_token config parameter is for use with Phantom instances. If both the token and username/password are given, the username and password will be used to authenticate to the Phantom instance.

Note that the IP (or name) being used must match the allowed IP in the remote Phantom instance's REST asset configuration.

For the update list action, if both, id and list_name parameters are given such that both of them point to different lists, then list_name parameter will be preferred and the action will update the list specified in the list_name parameter.

While running the create container action, it is mandatory to provide a label key in the container_json parameter. Also, to add the artifacts through this action, the artifacts should be provided as a list of artifact JSON object in the container_artifacts parameter. Each artifact JSON object should contain the following keys: cef, cef_types, data, description, end_time, ingest_app_id, kill_chain, label, name, owner_id, severity, source_data_identifier, start_time, tags, type. All other keys will be ignored. See Splunk Phantom Documentation for further details.

For the find artifacts action, the container_ids parameter will be considered only if the limit_search parameter is true. If any non-integer value is provided in the container_ids parameter, then all the non-integer values will be removed and the parameter will be updated accordingly.

In the add artifact action, to provide values containing double-quotes("), add a backslash(\) before the double-quotes. For example, {"X-Universally-Unique-Identifier":"test","Content-Type":"multipart/alternative; boundary=\"Apple-Mail=_0DA95D7E-B791-4751-8043-175949088A2C\">","Message-Id":""}

The actions, export container and import container, will fail with the error message like severity instance with name u'critical' does not exist, if the severity instances on the local phantom instance(the instance on which the action is being run) and the remote phantom instance(the instance which is configured in the asset) does not match.

The update artifact tags action will fail to add the tags for the artifact that does not contain label.

The deflate items action will be supported only if the phantom_server parameter(in the asset configurations) is configured to the same phantom instance on which the action is being run(or 0.0.0.0).

While running the add listitem action, if the list is updated through UI, then the user needs to save those changes before updating the list with the add listitem action, otherwise the changes made through UI will be overridden.

The actions, add artifact, find artifacts and add note, if no container_id is provided then it will take the current container(on which the action is being executed) as the container_id and the status will be reflected accordingly.

For information on how to obtain an authorization token, see Provisioning an Authorization Token in the Phantom REST Overview documentation.

If the value provided in the phantom_server configuration parameter is 0.0.0.0 then the test connectivity passes successfully and the actions will run on the current phantom instance, i.e., the server through which the app is being used.

See KB article 7 and KB article 16 on how to create and verify a valid HTTPS certificate for your Phantom instance.

For security reasons, accessing 127.0.0.1 is not allowed.
